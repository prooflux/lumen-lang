# Lumen MCP packaging

One-line install for the `lumen-mcp` server: a stdio Model Context Protocol server that exposes
a warm Lumen compiler (`lumen_check`, `lumen_fix`, `lumen_run`, `lumen_ir`, `lumen_explain`) to
any MCP-capable model.

## Quickstart

```sh
sh packaging/install.sh
```

This verifies `node >= 20`, runs `npm install` inside `seed/`, and prints the exact registration
command for your tool. It never registers the server itself — copy/paste the printed command.

Preview what it would do without touching anything:

```sh
sh packaging/install.sh --dry-run
```

## Register the MCP server

**Claude Code:**

```sh
claude mcp add lumen -- node /absolute/path/to/projects/lumen/seed/lumen_mcp.mjs
```

**Cursor** (`.cursor/mcp.json`):

```json
{
  "mcpServers": {
    "lumen": {
      "command": "node",
      "args": ["/absolute/path/to/projects/lumen/seed/lumen_mcp.mjs"]
    }
  }
}
```

Both commands are printed with the resolved absolute path by `install.sh`; use that output rather
than retyping the path.

## Skill / rules files

- `packaging/claude-skill/lumen/SKILL.md` — copy into `.claude/skills/lumen/SKILL.md` for a
  Claude Code skill that triggers on writing/checking/running `.lm` code.
- `packaging/cursor/lumen.mdc` — copy into `.cursor/rules/lumen.mdc` for the equivalent Cursor
  rule (auto-attaches to `**/*.lm` files).

## Hello world

```lumen
fn main(console: Console) -> Unit {
  console.print("hello, world\n")
}
```

Run it directly (no MCP needed) with the seed CLI:

```sh
cd seed && npm install
node run.mjs path/to/hello.lm
```

Or drive it through the MCP tools once registered: `lumen_run` with the source as the `source`
argument returns stdout, or structured diagnostics if it doesn't compile.

## Agent-focused tools (lumen_batch, lumen_profile, lumen_symbols)

In addition to the foundational tools (`lumen_check`, `lumen_fix`, `lumen_run`, `lumen_ir`,
`lumen_explain`), the MCP server exposes three tools optimized for agent workflows:

### lumen_batch

Compile up to 200 Lumen sources in one round-trip. Returns `{ok, diagnostics}` per source, in order.

**When to use:** when an agent needs to check the validity of multiple candidate programs (e.g., sampling N
variants from an LLM and filtering to only the ones that compile) without paying N separate round-trips.

**Example:**

```json
{
  "method": "tools/call",
  "params": {
    "name": "lumen_batch",
    "arguments": {
      "sources": [
        "fn main(console: Console) -> Unit { console.print(\"a\\n\") }",
        "fn main(console: Console) -> Unit { bad_syntax }",
        "fn double(n: Int) -> Int { return n * 2 } fn main(console: Console) -> Unit { console.print_int(double(21)) }"
      ]
    }
  }
}
```

Response:

```json
{
  "results": [
    { "ok": true, "diagnostics": [] },
    { "ok": false, "diagnostics": [{ "file": "...", "line": 1, "col": 44, "message": "error: expected '}', found 'EOF'" }] },
    { "ok": true, "diagnostics": [] }
  ]
}
```

### lumen_profile

Compile and run Lumen source with exact, reproducible cost accounting: total interpreter steps and per-function call counts (sorted by calls descending). Returns diagnostics if it doesn't compile.

**When to use:** when you need deterministic performance characterization of a Lumen kernel — the same program always reports identical step counts on every machine. Also reports which functions consume the most "CPU time" (steps), useful for bottleneck identification.

**Optional:** pass `report: true` to render a human-readable top-10-by-calls text report, generated by `profile_report.lm` (a Lumen program that sorts records and formats output, not a JS formatter).

**Example:**

```json
{
  "method": "tools/call",
  "params": {
    "name": "lumen_profile",
    "arguments": {
      "source": "fn fib(n: Int) -> Int { if n < 2 { return n } return fib(n - 1) + fib(n - 2) } fn main(console: Console) -> Unit { console.print_int(fib(10)) }",
      "report": true
    }
  }
}
```

Response (compact form):

```json
{
  "ok": true,
  "totalSteps": 12345,
  "functions": [
    { "name": "fib", "entry": 100, "calls": 177 },
    { "name": "main", "entry": 200, "calls": 1 }
  ],
  "report": "177 fib\n1 main\n"
}
```

### lumen_symbols

Outline a Lumen source's top-level functions for agent navigation: name, symbol-table entry address, source line, and the declaring line's text.

**When to use:** when an agent needs to understand the structure of a Lumen file without compiling/running it — extract the function signatures for code review, refactoring, or documentation.

**Example:**

```json
{
  "method": "tools/call",
  "params": {
    "name": "lumen_symbols",
    "arguments": {
      "source": "fn add(a: Int, b: Int) -> Int { return a + b }\nfn main(console: Console) -> Unit { console.print_int(add(2, 3)) }"
    }
  }
}
```

Response:

```json
{
  "ok": true,
  "symbols": [
    { "name": "add", "entry": 100, "line": 1, "signature": "fn add(a: Int, b: Int) -> Int {" },
    { "name": "main", "entry": 200, "line": 2, "signature": "fn main(console: Console) -> Unit {" }
  ]
}
```

## Where the oracle gates live

The compiler's correctness gates are the test suite in `seed/`:

```sh
cd seed && npm test
```

This runs, in order: `basics.mjs` (18 Lumen-mu programs compiled from source and executed),
`test.mjs` (conformance + safety-termination cases), `safety.mjs`, and `loop_test.mjs` (the
MCP/daemon round-trip loop, including the `lumend` incremental-edit daemon's
edit-to-diagnostic latency gate, p50 < 5ms). All of it runs against the single warm
`compiler_core.mjs` instance that `lumen_mcp.mjs` also embeds — the tests and the MCP server share
the same compiler, so a green `npm test` is the real oracle for "does the packaged server work".
